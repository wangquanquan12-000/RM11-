## 一、产品审查结论（@产品）

### 1.1 逻辑审查

| 检查项 | 现状 | 结论与风险 |
|--------|------|------------|
| **登录 → 主应用** | Cookie 7 天 + session_state.auth_logged_in_user；未登录展示登录/注册页 | ✅ 逻辑清晰；退出时清除 Cookie 与 session，一致 |
| **生成用例主流程** | ① Quip URL → ② 账号/模型（可折叠）→ ③ 导出方式 → ④ 执行；成功后写 app_last_run / app_last_demand_* | ✅ 步骤明确；⚠️ 未保存凭证时仅提示「请填写」，未在 UI 显式区分「未配置」与「已配置未勾选保存」 |
| **自动归档** | 勾选后从 result_str 解析表格，追加到 full_regression | ✅ 与 memory_store 约定一致；⚠️ 解析失败时无用户可见提示 |
| **编辑 Agent** | 从 agents.yaml 读入，表单编辑后整份写回 | ✅ 与配置驱动方向一致；⚠️ 无「未保存离开」提示，误关页会丢改 |
| **项目记忆** | 搜索 → 列表展示 → 删除；导入（文件夹/单文档/测试用例） | ✅ 导入路径清晰；⚠️ 批量导入进度条与 status_text 在 rerun 后可能闪烁，可考虑统一进度组件 |
| **文档问答** | 来源（记忆/粘贴/Quip）→ 加载 doc_context → 对话 append 到 app_doc_chat_messages | ✅ 状态在 session；⚠️ 无「当前文档标题」在对话区展示，长对话时易忘上下文 |
| **状态命名** | app_last_run、app_doc_chat_messages 等已带 app_ 前缀 | ✅ 与现有规范部分对齐；❌ 尚未引入 current_task_context 与 module_id 标准化，见架构规范 |

**逻辑优化建议（纳入方案）**  
- 账号/模型区：增加「未保存凭证」与「已保存」的显式状态区分（如徽标或短文案）。  
- 自动归档：解析失败时给出 toaster 或 st.warning，避免静默失败。  
- 编辑 Agent：离开页面前若有未保存修改，给出确认（或统一「有未保存修改」提示）。  
- 文档问答：在对话区上方固定展示「当前文档：xxx」，便于上下文感知。

### 1.2 UI 审查

| 检查项 | 现状 | 结论与优化建议 |
|--------|------|----------------|
| **整体布局** | 主区 max-width 920px；侧栏仅登录态显示用户与退出 | ✅ 主次清晰；建议侧栏可配置「工作台入口」占位，为后续多模块预留 |
| **步骤标签** | ① ② ③ ④ + step-label 样式 | ✅ 层级清楚 |
| **Tab** | 生成用例 / 编辑 Agent / 项目记忆 / 文档问答，文案来自 ui_texts.yaml | ✅ 文案配置化；❌ Tab 列表仍在代码中写死，不符合「工作台模块配置驱动」 |
| **模型选择** | GEMINI_MODELS 在代码中写死元组列表 | ❌ 违反配置驱动；必须改为从 config（如 models.yaml）读取 |
| **错误与成功反馈** | st.error / st.success / st.warning | ✅ 有反馈；建议长时间运行（如流水线）增加「可取消」或至少「运行中」的明确状态 |
| **Expander 使用** | 账号/模型、步骤详情、结果全文 均用 expander | ✅ 减少首屏信息量；建议「输出与下载」在成功后可默认展开或更突出 |
| **登录/注册** | 双 Tab + 验证码 + 万能码 | ✅ 流程完整；验证码为纯数字，可考虑提示「万能码」仅内部使用 |

**UI 优化建议（纳入方案）**  
- 运行中：主按钮区域显示「运行中…」+ 不可重复点击，或使用统一进度组件（见架构规范）。  
- 输出与下载：成功后将「下载 Excel / 打开 Quip / 打开 Sheets」提升为更显眼的区块（如置顶或高亮框）。  
- 为后续测试/埋点预留：关键操作（生成用例、保存配置、导入记忆、发送对话）使用稳定的 key 或 data-*，便于测试人员写自动化与埋点。

## 二、产品与编程共识方案

以下为 @产品 与 @编程 讨论后的共识，作为本期实现的**产品方案**。

### 2.1 本期目标（必须完成）

1. **配置驱动补齐**  
   - 模型列表从 `config/models.yaml`（或等效）读取，删除代码中的 `GEMINI_MODELS` 写死。  
   - 工作台模块列表从 `config/workbench_apps.yaml` 读取，Tab 或导航项由配置驱动；现有 4 个功能作为 4 个模块在配置中声明。

2. **工作台形态**  
   - 主区为「工作台」：顶部或侧栏为模块入口（动态 Tab 或 st.navigation），当前 4 个模块（Quip 转用例、编辑 Agent、项目记忆、文档问答）作为独立模块渲染，互不写死依赖。

3. **统一状态总线**  
   - 引入 `st.session_state["current_task_context"]` 结构（含 module_id, task_id, input_summary, output_summary, status, started_at）；  
   - 现有 `app_last_run` 等迁入「按 module_id 命名」的约定（如 app_quip_to_cases_last_run），并在文档中注明 key 表。

4. **公用组件**  
   - 至少抽出 3 类：**Agent 执行进度/状态**、**文件上传器**、**日志/终端展示**，放入 `ui_components/` 或 `app_ui_components.py`，并在至少 2 处复用（如生成用例 + 项目记忆导入）。

5. **产品体验优化（本期必做）**  
   - 账号/模型：显式区分「未保存」与「已保存」状态。  
   - 自动归档失败：解析失败时给出明确提示。  
   - 编辑 Agent：有未保存修改时离开页给出提示或确认。  
   - 文档问答：对话区上方展示「当前文档：xxx」。  
   - 运行中：主操作不可重复提交，并显示运行中状态（或统一进度组件）。

### 2.2 本期不做（留给后续迭代）

- 真正的「插件」热插拔（如从目录扫描 .py 自动注册）。  
- 测试人员角色的规则与用例库（测试加入后再定协作流程与用例归属）。  
- 大 UI 视觉改版（仅做上述逻辑与结构优化）。

### 2.3 与测试的协作预留

- 关键操作使用稳定、可选的 key 或 data 属性，便于后续测试人员编写自动化与用例。  
- 产品验收标准见下文「实施清单」中的验收项；测试人员加入后，可在同一文档下补充「测试用例清单」与「回归范围」。

## 三、给 @编程 的实施清单

请按以下顺序实现，并在完成后自测 + 对照《ui-architecture-spec.md》检查清单。

### Phase 1：配置与工作台骨架

1. **新增 config/models.yaml**  
   - 结构：模型 key、展示名、是否默认；与当前 GEMINI_MODELS 等价。  
   - 修改 app_ui：模型下拉框从该配置读取，删除硬编码列表。

2. **新增 config/workbench_apps.yaml**  
   - 定义 4 个模块：id、名称、顺序、enabled；id 与现有 Tab 对应（如 quip_to_cases, agents, memory, chat）。  
   - 主应用入口：根据该配置渲染 Tab 或 st.navigation，每个 id 对应一个渲染函数（现有 4 块逻辑迁入 4 个函数）。

3. **统一状态总线**  
   - 在合适位置初始化/更新 `session_state["current_task_context"]`（结构见架构规范）；  
   - 将现有 app_last_run、app_last_demand_snippet、app_last_demand_full 迁入带 module_id 的 key（如 app_quip_to_cases_last_run），并保留兼容读取（过渡期可双写或兼容旧 key）。  
   - 在 `docs/` 或代码注释中维护「session_state key 一览表」。

### Phase 2：公用组件

4. **ui_components 或 app_ui_components.py**  
   - **进度/状态组件**：入参（当前步、总步、状态文案），供「生成用例」与「项目记忆批量导入」使用。  
   - **文件上传组件**：类型/大小限制，返回统一结构；供项目记忆「上传文件」等使用。  
   - **日志/终端组件**：接收行列表或流，可折叠、可复制；供「查看 Agent 沟通过程」及后续模块复用。  

5. **替换内联实现**  
   - 生成用例、项目记忆中的进度与展示改为调用上述组件；至少 2 处复用进度、1 处复用上传、1 处复用日志。

### Phase 3：产品体验优化

6. **账号/模型**：根据是否已保存凭证显示不同文案或徽标（如「已保存 ✓」与「未保存，请填写并保存」）。  
7. **自动归档**：若解析表格失败，st.warning 提示「归档失败：未解析到有效表格」。  
8. **编辑 Agent**：在 agents 页有编辑未保存时，若用户切换 Tab 或刷新，给出 st.warning 或确认（实现方式可自选，以不丢改为准）。  
9. **文档问答**：在对话列表上方固定一行「当前文档：{标题}」或「无文档」。  
10. **运行中**：点击「生成测试用例」后按钮禁用或变为「运行中…」，避免重复提交；可与进度组件结合。

### Phase 4：自测与文档

11. **回归**：现有 4 个 Tab 的完整流程（生成用例、编辑 Agent、项目记忆导入与搜索、文档问答）行为与当前一致，无功能回退。  
12. **文档**：更新或补充 `docs/ui-architecture-spec.md` 中的「session_state key 一览表」；在 `docs/manual_test_checklist.md` 中增加「工作台模块切换」「模型来自配置」等检查项。
