# 代码优化方案 · 架构师视角

> **文档用途**：从架构与工程角度梳理可落地的代码优化思路，供迭代排期与 Code Review 参考。
>
> **创建日期**：2025-03-02

---

## 一、架构层面

### 1.1 消除 Crew 双路径，统一为配置驱动

**现状**：
- `crew_test.py` 存在两套执行路径：
  - `_get_crew()`：硬编码 Agent / Task（约 300 行）
  - `_run_crew_sequential()`：基于 `config/agents.yaml` 的配置驱动
- `run_pipeline` 按 `use_config` 分支选择其一，增加心智负担与回归风险

**建议**：
- 逐步废弃 `_get_crew()`，在 `run_pipeline` 中仅保留 `_run_crew_sequential` 路径
- 若需兜底，可将硬编码 Agent 作为「默认配置」的来源，而非独立逻辑
- **收益**：单一执行路径、易维护、与 `config/agents.yaml` 编辑能力对齐

---

### 1.2 crew_test 职责拆分

**现状**：`crew_test.py` 承载 Quip API、Crew 构建与执行、需求加载、表格解析与导出、项目记忆、脱敏等，单文件职责过重。

**建议**：按领域拆分模块：

```
services/
├── quip_client.py        # Quip API 封装（拉取、文件夹遍历）
├── crew_orchestrator.py  # Crew 构建 + 顺序执行
├── export_service.py     # Excel / Quip / Sheets 导出
├── document_filter.py    # is_product_requirement_doc、doc_filter
└── project_context.py    # load_project_memory, get_project_context_for_agent
```

- 保留 `crew_test.py` 作为上层入口，内部调用各 service
- **收益**：职责清晰、单测可覆盖、扩展新数据源/导出渠道更简单

---

### 1.3 app_ui 瘦身与 UI 分层

**现状**：`app_ui.py` 约 1569 行，混入配置加载、UI 渲染、部分业务逻辑，可读性与维护成本偏高。

**建议**：

| 优化项 | 做法 |
|--------|------|
| 配置加载 | 抽离到 `config_loader.py`，统一 `_load_defaults`、`_load_models`、`_load_version`、`_load_workbench_apps` 等 |
| 各模块渲染 | 将 `_render_module_*` 拆分到 `pages/` 或 `app_ui_components.py` 子模块，主文件仅做路由 |
| 设计系统 | 将 100+ 行 CSS 移到 `static/styles.css` 或独立文件，用 `st.markdown(open(...))` 注入 |
| 文案 | 已使用 `config/ui_texts.yaml`，确保所有提示语从文案库读取，避免硬编码 |

**收益**：主入口简洁、新模块易接入、样式与逻辑解耦。

---

## 二、代码质量

### 2.1 消除重复与魔法值

**现状**：
- `_DEPRECATED_GEMINI_MODEL_MAP` 在 `crew_test.py` 与 `app_ui.py` 各有一份
- `DEFAULT_REQUIREMENT_FILE`、`OUTPUT_DIR`、`CONFIG_DIR` 等在多处散落

**建议**：
- 在 `config/constants.py` 或 `config/models.yaml` 中集中定义弃用模型映射、默认路径
- 各模块从统一入口引用，避免重复实现

---

### 2.2 异常与重试策略

**现状**：Quip 请求有简单重试，但 Gemini、Sheets 等缺少统一 retry；部分 `except Exception` 过于宽泛。

**建议**：
- 引入 `tenacity` 等库，对 API 调用做统一重试装饰（如 Quip 503、Gemini 限流）
- 区分业务异常（凭证缺失、校验失败）与系统异常（网络、超时），分别处理与展示
- 业务层可考虑 `Result[T, Error]` 风格，减少裸异常传播

---

### 2.3 类型与接口契约

**现状**：已有 `TypedDict`（如 `QuipToCasesResult`），但部分函数仍用 `dict[str, Any]` 或裸 `dict`。

**建议**：
- 为 Pipeline 输入输出定义 TypedDict / dataclass，如 `PipelineInput`、`PipelineOutput`、`StepOutput`
- 导出、拉取等关键路径的返回类型显式声明，便于静态检查与重构

---

## 三、性能与可维护性

### 3.1 配置与依赖加载

**现状**：YAML 每次调用重新读取，无缓存；`load_agents_config`、`_load_doc_filter_config` 等在流程中多次调用。

**建议**：
- 对「启动后不变」的配置做轻量缓存（如 `@lru_cache` 或带文件 mtime 的 dict）
- 若需热更新，可通过版本号或显式刷新接口触发失效

---

### 3.2 LLM / Crew 初始化

**现状**：`_get_crew()` 有全局 `_crew` 缓存，但 `_run_crew_sequential` 每次新建 LLM、Agent；执行 4 个 task 时重复初始化较多。

**建议**：
- 对「模型 + 配置不变」的场景，考虑 Agent / Crew 实例级缓存
- 以配置 hash 或版本号控制缓存失效，减少重复初始化

---

### 3.3 长流程断点续跑

**现状**：四 Agent 顺序执行，中途失败时已完成的 step 未持久化，用户需全部重跑。

**建议**：
- 每步完成后将 `step_outputs` 写入临时文件或 `st.session_state`
- 支持「从断点续跑」或至少展示已完成部分的中间结果，降低重试成本

---

## 四、安全与健壮性

### 4.1 凭证与敏感数据

**现状**：已使用 `credential_store`、`st.secrets` 等，方向正确。

**建议**：
- 避免将完整 token 放入 `st.session_state` 并随错误信息打印
- 检查 `st.error`、`st.warning`、日志中是否有 token 泄露风险

---

### 4.2 输入与输出校验

**现状**：`_sanitize_cell_for_excel` 已做 CSV/Excel 公式注入防护，符合规范。

**建议**：
- 对 `load_demand_from_quip`、`parse_uploaded_files` 的入参做基础校验（长度、编码、格式）
- 大模型输出解析失败时，返回明确错误提示，避免白屏或静默失败

---

## 五、实施优先级建议

| 优先级 | 优化项 | 预估工作量 | 收益 |
|--------|--------|------------|------|
| **P0** | 消除 Crew 双路径，统一配置驱动 | 中 | 降低维护成本、避免双轨回归 |
| **P0** | crew_test 拆分为 services 子模块 | 中 | 可测试性、可扩展性 |
| **P1** | app_ui 拆分为配置加载 + 组件 + 路由 | 中 | 可读性、多人协作 |
| **P1** | 统一弃用模型映射与常量 | 小 | 消除重复、易升级 |
| **P2** | 配置缓存、Agent 实例缓存 | 小 | 启动与执行性能 |
| **P2** | tenacity 重试与异常分层 | 小 | 稳定性、可观测性 |
| **P3** | 长流程断点续跑 | 大 | 用户体验、成本节约 |

---

## 六、小结

当前架构已具备较好的分层（`pipeline_service`、`config/agents.yaml`、`ui_texts.yaml`），主要优化空间集中在：

1. **单一执行路径**：废弃硬编码 Crew，仅保留 `agents.yaml` 驱动
2. **职责拆分**：`crew_test` → 多 service 模块；`app_ui` → 配置 + 组件 + 路由
3. **去重与收敛**：常量、弃用映射、配置加载集中管理
4. **健壮与可观测**：重试、异常分层、类型契约

建议按 P0 → P1 → P2 顺序推进，在不破坏现有功能的前提下逐步提升可维护性与扩展性。
