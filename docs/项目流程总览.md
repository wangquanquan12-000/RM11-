# 用例工坊 · 项目流程总览

> **文档用途**：从产品+工程双视角，串起当前版本「用例工坊」的整体项目流程，帮助新成员在 5 分钟内搞清楚：用户从哪里进来、做什么操作、哪些模块在背后配合、核心数据如何流转。
>
> **关联文档**：`四Agent提示词与任务编排-规格说明.md`、`Agent知识库-PRD.md`、`项目记忆-RAG与上下文管理-PRD.md`、`项目记忆-重新规划实施方案.md`、`项目记忆摘要修复-PRD与开发文档.md`、`设计图导入-PRD.md`

## 一、用户视角的整体路径

> 站在测试同学视角，**一次完整的「需求 → 用例」工作流**。

1. **登录平台**
   - 打开 Streamlit 应用，登录/注册进入工作台首页。
   - 看到顶部或侧栏的模块入口：`生成用例`、`项目记忆`、`文档问答`、`需求风险分析`、`设置` 等。

2. **准备需求文档（入口：生成用例模块）**
   - 选择「生成用例」主模块。
   - 通过两种方式之一提供需求内容（**已完全去除 Quip / 外部文档模式**）：
     - 上传 `.md` / `.docx` 等文件；
     - 直接在文本框中粘贴 PRD 全文。
   - 可选：上传既有全回归用例表格（Excel），供 Agent 参考已有覆盖范围。

3. **一键生成测试用例**
   - 用户点击「生成测试用例」，后端触发 **四 Agent 协作流水线**（见第三节）：
     - 1 号：文档分析师做风险评估；
     - 2 号：测试点拆解师结构化测试点；
     - 3 号：测试开发 Agent 产出 9 列标准用例表；
     - 4 号：QA Reviewer 做表格级审查与修正。
   - 用户在 UI 中看到生成完成提示，并可：
     - 在线预览用例表；
     - 一键导出 Excel（单元格内 `<br>` 支持有效换行）。

4. **查看与管理生成历史**
   - 在生成用例模块中，可通过历史记录查看近期生成的用例文件（标题为「需求大标题 + 测试用例」）。
   - 支持：
     - 下载历史用例 Excel；
     - 删除不再需要的历史记录。

5. **维护项目级背景知识（入口：项目记忆模块）**
   - 用户切换到「项目记忆」Tab，维护**跨需求、跨版本的长期项目背景**，包括：
     - 粘贴导入需求文档（`source_type = manual`）；
     - 上传或粘贴全回归测试用例（`source_type = test_cases`）；
     - 上传并解析设计图（`source_type = design_mockup`），让 Agent 理解真实 UI 布局与交互；
     - 从最近一次运行中追加摘要（`source_type = run_summary`）。
   - 所有导入记录在时间轴中展示，并由 Librarian Agent 自动生成 150 字摘要，方便快速回顾。

6. **基于项目记忆做文档问答（入口：文档问答模块）**
   - 在「文档问答」模块中，用户可以基于：
     - 已导入的全部项目记忆（需求 + 全回归 + 设计图描述）；
     - 或当前粘贴的一份文档，
     提问「总结/对比/关联」类问题（例如：该需求与之前某次改版的差异、隐藏风险点等）。

7. **独立生成风险报告（入口：需求风险分析模块）**
   - 当用户只想做风险分析而不是生成用例时，可以进入「需求风险分析」模块：
     - 选择文档来源（粘贴 / 项目记忆中选择近期文档）；
     - 一键产出表格化风险报告，用于评审会。

8. **设置与版本校验（入口：设置模块）**
   - 在「设置」中配置：
     - Gemini API Key；
     - 默认模型；
     - 本地 workspace 路径（输出文件目录）等。
   - 侧栏展示当前应用版本号（来自 `config/version.yaml`），用于上线后校验「代码是否成功更新」。

## 二、模块视角：当前工作台组成

> 从「工作台模块」而非「页面」的维度，梳理当前项目的主干模块。

1. **生成用例模块**
   - 职责：将单份 PRD + 可选全回归用例 → 高质量测试用例 Excel。
   - 输入：
     - 需求文档：上传/粘贴；
     - 可选：全回归用例表（本次需求相关）。
   - 输出：
     - Excel 测试用例表（固定 9 列格式）；
     - 运行记录（标题、时间戳、输出路径），写入本地 JSON 供刷新后回显。

2. **项目记忆模块**
   - 职责：沉淀项目级长期记忆，作为多 Agent 的统一背景。
   - 能力：
     - 导入需求（manual）；
     - 导入全回归用例（test_cases）；
     - 导入设计图描述（design_mockup）；
     - 维护 `project_memory.md`「项目记忆摘要」；
     - 触发 Agent 知识库刷新；
     - 展示导入历史时间轴 + Librarian 摘要。

3. **Agent 知识库（挂在项目记忆模块下）**
   - 职责：把零散的记忆条目（需求 + 用例 + 运行摘要）消化为结构化 Markdown 知识库 `agent_knowledge.md`。
   - 特点：
     - 由 `agent_knowledge_service.py` 维护；
     - 支持自动过期（≥7 天）与手动刷新；
     - 生成用例时，四 Agent 优先参考知识库，而非重复阅读大量原文。

4. **文档问答模块**
   - 职责：为测试/产品提供一个「问文档」的对话界面。
   - 文档来源：
     - 项目整体记忆（全部需求文档）；
     - 手动粘贴单份文档。
   - 输出：自然语言回答，辅助理解和评审。

5. **需求风险分析模块**
   - 职责：对单份文档产出表格化风险报告，不参与四 Agent 用例流水线。
   - 输出：可直接用于评审会的风险列表、优先级建议。

6. **设置/版本模块**
   - 职责：集中管理密钥、模型、workspace 路径等全局配置，并在侧栏展示版本号。

## 三、Agent 与上下文的整体流程

> 重点说明：四 Agent 如何拿到上下文、如何接力、最终如何落到 Excel。

1. **上下文拼装（project_context）**
   - 基础来源：
     - `project_memory.md`：人工维护的项目背景摘要；
     - `fambase_modules.yaml`：主模块/子模块与用例编号（如 `LIV-0001`）；
     - `agent_knowledge.md`：Agent 知识库（全局理解）。
   - 增量来源（可选，取决于 memory_store 是否可用）：
     - 最近导入的需求与全回归用例（`get_recent_for_agent`）；
     - 设计图解析出来的 UI 描述文本。
   - 形成规则：
     - 有项目记忆存储时：拼接「项目记忆 + 模块定义 + Agent 知识库 + 近期记忆」；
     - 无项目记忆存储时（如云端无 sqlite3 且 JSON 后端也异常）：退化为「项目记忆 + 模块定义」，**生成用例仍可用**。

2. **四 Agent 协作流水线**
   - **Document Analyst（1 号）**：基于 `{prd_content}` + project_context 找出需求中的风险与模糊点，输出风险报告。
   - **Requirements Analyst（2 号）**：吸收风险报告与文档本身，拆解出模块化测试点大纲。
   - **Case Agent（3 号）**：基于测试点大纲，产出标准化测试用例表（9 列：ID / 模块 / 测试项 / 优先级 / 前置条件 / 操作步骤 / 预期结果 等）。
   - **QA Reviewer（4 号）**：审查并修正用例表，去除重复/低价值用例，保证覆盖关键链路与极端场景。

3. **Excel 导出与安全处理**
   - Markdown 表格 → 内部表结构 → Excel：
     - 支持 `<br>` 转换为单元格内换行；
     - 对以 `= + - @` 开头的内容自动加单引号，防止 CSV/Excel 公式注入。

## 四、项目记忆与存储的安全设计（高层总结）

> 源自 `项目记忆-重新规划实施方案.md`，简化版说明，便于快速理解为什么不会再因为 sqlite3 崩溃。

1. **项目记忆是可选增强，不是硬依赖**
   - 即使 memory_store 完全不可用，核心功能（生成用例、文档问答、风险分析）也要能正常工作。

2. **存储后端可插拔**
   - **SqliteBackend**：本地/支持 sqlite3 的环境使用，存储在 `config/memory.db`。
   - **JsonFileBackend**：云端无 sqlite3 时，自动降级为 `config/memory_entries.json`。
   - 两个后端对外接口一致，UI 与上层逻辑无感知。

3. **惰性初始化与多层兜底**
   - 模块顶层不再直接 connect DB；首次真实读写才连接/建表。
   - `app_ui.py` 与 `crew_test.py` 中对 `memory_store` 的 import 均有 try-except 保护。

4. **哈希去重与 Librarian 摘要**
   - 所有导入条目统一走 `add_entry_with_dedup`，使用 SHA-256 避免重复入库；
   - 新增或更新的条目由 Librarian Agent 生成 150 字摘要，作为导入历史中的概览。

## 五、未来扩展的预留点（简要）

1. **Gemini Context Caching**
   - 当前：四 Agent 直接使用拼好的 project_context 字符串；
   - 未来：可将 project_context 作为 Cached Content 交给 Gemini，所有 Agent 请求复用同一份缓存，降低 Token 与延迟。

2. **更多来源的项目记忆**
   - 如：Jira 工单、飞书群聊纪要、日志提炼等，均可以通过独立模块导入到 memory_store 中，复用相同的去重与摘要机制。

3. **更细粒度的用例回溯**
   - 未来可以在用例表中反向标记其依赖的需求条目、设计图、风险报告条目，形成闭环追踪；当前版本暂不实现。
