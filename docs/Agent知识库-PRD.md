# Agent 知识库 · 独立功能 PRD

> **文档用途**：供开发实施「Agent 知识库」能力。让 4 个 Agent 基于一份持续维护的知识库进行推理，而非每次生成时重喂原始需求文档与全回归用例。
>
> **创建日期**：2025-03-02

---

## 一、背景与目标

### 1.1 背景

当前：需求文档与全回归用例导入到 `config/memory.db`，生成用例时 `get_project_context_for_agent()` 从 `project_memory.md` + `memory_entries`（近期需求 + 全回归）拼成 project_context 注入 4 个 Agent。

问题：每次生成都「喂」大量原文，上下文冗长且重复；不符合「Agent 形成长期记忆」的预期。

### 1.2 目标

- 新增统一知识库文件：`config/agent_knowledge.md`。
- 系统对「全部需求文档 + 全回归用例 + 运行产出」做总结式消化，产出结构化 Markdown。
- 4 个 Agent 生成时只读 `project_memory.md + agent_knowledge.md`，不再重喂原文。
- 与 `agent_summary` 独立；知识库为更高层「全局理解」。

---

## 二、范围与非目标

### 2.1 范围

- 新增 `config/agent_knowledge.md`，覆盖式更新。
- 新增知识库构建服务：从 memory_store + project_memory 拉取，调用 LLM 总结，覆盖写入。
- 调整 `get_project_context_for_agent()` 数据来源，使 4 个 Agent 默认使用知识库。
- 在「项目记忆」页增加：知识库最后更新时间、手动「刷新知识库」按钮、过期提示。
- **更新机制**：支持两种方式：
  - **自动更新**：默认每 7 天自动执行一次知识库构建。
  - **手动更新**：用户点击「刷新知识库」按钮立即执行。

### 2.2 非目标

- 不改变现有导入流程。
- 不改变 `get_project_context_for_agent` / `get_recent_for_agent` 函数签名。
- 不实现复杂的调度器（如 CRON）；自动更新通过「启动/定时检查 + 距上次更新 ≥ 7 天则触发」即可。

---

## 三、数据与存储设计

### 3.1 原始数据层（沿用）

- SQLite：`config/memory.db` → `memory_entries`（`manual` / `test_cases` / `run_summary` 等）。
- Markdown：`config/project_memory.md`（手工项目背景）。

### 3.2 知识库层（新增）

- 文件：`config/agent_knowledge.md`，由系统覆盖式写入。

建议结构：

```markdown
# Agent 知识库（Fambase）

## 一、产品域概览
- 项目简述与关键域（群组聊天、群内Live、Coins、礼物打赏、V3后台...）

## 二、核心模块与职责
### 群组聊天
- 核心规则摘要、常见风险模式

### 群内 Live 直播
- 核心规则摘要、常见风险模式

### Coins 与礼物打赏
- 计费与风控要点、与全回归用例覆盖关系

## 三、需求文档变更脉络（按时间）
- [日期] 文档《xxx》：新增/修改 xxx，影响模块：...

## 四、全回归用例覆盖地图
- 模块 A：已覆盖场景类型
- 模块 B：覆盖薄弱区域

## 五、历史运行经验
- 常见缺陷模式、已修正典型问题
```

### 3.3 与 agent_summary 关系

| 对象 | 粒度 | 读者 | 用途 |
|------|------|------|------|
| agent_summary | 单条导入的局部摘要 | 人 | 导入历史页查看 |
| agent_knowledge.md | 全局知识库 | Agent + 人 | 长期记忆与推理背景 |

## 四、知识库更新机制

### 4.1 两种更新方式

| 方式 | 说明 | 触发 |
|------|------|------|
| **自动更新** | 默认每 7 天自动执行一次 | 应用启动时或后台检测：若距离上次成功更新时间 ≥ 7 天，则自动执行知识库构建 |
| **手动更新** | 用户主动刷新 | 在「项目记忆」页点击「刷新知识库」按钮 |

- 自动更新与手动更新执行同一套构建逻辑。
- 任意一种方式成功后，更新「知识库最后更新时间」，7 天计时重置。

### 4.2 自动更新实现要点

- 记录上次成功更新时间：可存于 `config/agent_knowledge_meta.json`（如 `{"last_updated": "2025-03-02T10:00:00"}`）或使用 `agent_knowledge.md` 的 mtime。
- 触发时机（二选一或组合）：
  - **方案 A**：Streamlit 应用每次启动时检查，若 ≥ 7 天则后台或前台执行一次构建。
  - **方案 B**：在用户进入「项目记忆」页时检查，若 ≥ 7 天则提示「知识库已过期，正在自动更新…」并执行。
- 自动更新过程中若有 UI，可显示 spinner；失败时记录错误，不阻塞主流程，下次手动或自动再试。

### 4.3 手动更新

- 在「项目记忆」页的「Agent 知识库」区块提供「刷新知识库」按钮。
- 点击后立即执行构建，显示 spinner，完成后更新展示；失败则提示错误并保留旧知识库。

## 五、知识库构建流程（逻辑）

1. **收集原始数据**：从 `memory_entries` 读需求、全回归、run_summary（可选），按时间排序。
2. **预处理**：按模块/类型聚合，控制输入长度（如 30k–50k 字符）。
3. **构造 Summarize Prompt**：要求 LLM 按 3.2 结构输出 Markdown。
4. **调用 LLM**：使用与流水线一致的 Gemini 模型与 Key，失败可重试 2–3 次。
5. **写入 agent_knowledge.md**：成功则覆盖写入；失败则保留旧文件并在 UI 提示。

---

## 六、与 Agent 调用的关系

### 6.1 get_project_context_for_agent 调整

- **目标逻辑**：`project_context = project_memory.md + agent_knowledge.md`
- 若 `agent_knowledge.md` 不存在或为空，则退回现有逻辑（`get_recent_for_agent` 等）。

### 6.2 四个 Agent

- 通过 `{project_context}` 注入同一份知识库；task4 若尚未包含需补齐。

---

## 七、UI 交互（项目记忆页）

- 「Agent 知识库」区块：
  - 显示「知识库最后更新时间：YYYY-MM-DD HH:MM」或「尚未生成」。
  - 按钮「刷新知识库」（手动更新）。
  - 若距今 ≥ 7 天未更新，显示提示「知识库已超过 7 天未更新，建议点击【刷新知识库】或等待自动更新。」

---

## 八、异常与边界处理

| 场景 | 处理 |
|------|------|
| memory_entries 为空 | 生成最小模板知识库，标注「暂无导入数据」 |
| LLM 超时/429/503 | 提示「刷新失败，请稍后重试」，不覆盖旧知识库 |
| 写文件失败 | st.error 提示，不覆盖旧知识库 |
| 自动更新失败 | 静默记录，不阻塞；下次手动或自动再试 |

## 九、验收标准

| 编号 | 验收项 |
|------|--------|
| AC1 | 「项目记忆」页有「Agent 知识库」区块，显示最后更新时间或「尚未生成」。 |
| AC2 | 点击「刷新知识库」可手动构建并覆盖 `agent_knowledge.md`。 |
| AC3 | **自动更新**：距上次成功更新 ≥ 7 天时，系统自动执行一次知识库构建（启动或进入项目记忆页时触发）。 |
| AC4 | 4 个 Agent 的 project_context 主要来自 `project_memory.md + agent_knowledge.md`（知识库存在时）。 |
| AC5 | 知识库不存在或构建失败时，退回现有逻辑，不报错。 |
| AC6 | 自动/手动更新失败时，有明确提示，不覆盖旧知识库。 |
