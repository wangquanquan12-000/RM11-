# 项目记忆 · 重新规划实施方案

> **背景**：上一版实现硬依赖 `sqlite3`，云端环境（Streamlit Cloud）缺失该模块导致整站白屏。本文档重新规划项目记忆的实现逻辑，从架构层面消除此类风险。
>
> **关联文档**：`项目记忆-RAG与上下文管理-PRD.md`、`项目记忆-RAG与上下文管理-开发实施文档.md`

---

## 一、事故复盘

### 1.1 崩溃链路

```
Streamlit 启动
  └─ app_ui.py 第 35 行：from memory_store import (...)
       └─ memory_store.py 第 9 行：import sqlite3   ← 云端无此模块
            └─ ImportError  ← 整个 app_ui.py 无法加载
                 └─ Streamlit 白屏（核心功能全部不可用）
```

### 1.2 根因分类

| 层级 | 问题 |
|------|------|
| **依赖层** | `sqlite3` 是 CPython 标准库，但 Streamlit Cloud 的 Python 运行时可能裁剪了该模块 |
| **模块层** | `memory_store.py` 在顶层 `import sqlite3`，模块 import 即触发异常 |
| **调用层** | `app_ui.py` 虽有 try-except 保护，但 `memory_store` 内部的 import 失败先于 try-except 执行 |
| **架构层** | 项目记忆被当做硬依赖而非可选增强，无 fallback 后端 |

### 1.3 当前降级措施

开发 Agent 已将 `app_ui.py` 中 `memory_store` 的 import 包裹在 try-except 里，设置 `MEMORY_AVAILABLE = False`。效果：核心功能恢复，但项目记忆功能完全不可用。

## 二、新设计原则（铁律）

| 编号 | 原则 | 说明 |
|------|------|------|
| R1 | **项目记忆是可选增强** | 任何情况下，项目记忆不可用 ≠ 生成用例不可用 |
| R2 | **存储后端可插拔** | sqlite3 是首选，缺失时自动降级到 JSON 文件后端 |
| R3 | **惰性初始化** | 模块顶层只做 class 定义和常量，DB 连接 / 文件读取推迟到首次真正读写 |
| R4 | **调用方多层防御** | 每个 `from memory_store import` 都必须有 try-except 兜底 |
| R5 | **memory_store 自身不抛 ImportError** | 后端选择在模块内部完成，对外永远是可 import 的 |

## 三、可插拔后端架构

### 3.1 整体结构

```
memory_store.py
├── IMemoryBackend（协议/接口）
│     ├── add_entry(...)
│     ├── add_entry_with_dedup(...)
│     ├── search(keyword, limit)
│     ├── list_recent(limit)
│     ├── list_import_history(limit)
│     ├── delete_entry(entry_id)
│     ├── get_entry_content(source_type, source_id)
│     ├── update_agent_summary(entry_id, summary, status)
│     ├── get_recent_for_agent(limit, ...)
│     ├── get_all_demands_full_for_chat(limit, ...)
│     └── list_for_browse(source_type_filter, limit)
│
├── SqliteBackend（实现 IMemoryBackend）
│     └── 内部惰性初始化 sqlite3 连接
│
├── JsonFileBackend（实现 IMemoryBackend）
│     └── 基于 config/memory_entries.json 读写
│
└── 模块级自动选择
      try:
          import sqlite3 → _backend = SqliteBackend()
      except ImportError:
          _backend = JsonFileBackend()
```

### 3.2 后端能力对比

| 能力 | SqliteBackend | JsonFileBackend |
|------|:---:|:---:|
| CRUD | 全量 | 全量 |
| SHA-256 去重 | 支持 | 支持 |
| 关键词搜索 | SQL LIKE（多词 AND） | 遍历匹配（性能略低，数据量小可接受） |
| 并发写入 | sqlite 自带锁 | 文件锁（简单实现，Streamlit 单用户场景足够） |
| 索引 | B-tree | 无（线性扫描） |
| 迁移性 | 需 sqlite3 | 纯 JSON，任何环境可用 |

### 3.3 对外接口完全不变

```
对外函数签名保持一致：
  add_entry(source_type, content, source_id, title, summary) → int
  add_entry_with_dedup(source_type, content, ...) → tuple[int, str]
  search(keyword, limit) → list[dict]
  delete_entry(entry_id) → bool
  ...

调用方（app_ui.py、crew_test.py、agent_knowledge_service.py）零改动。
```

### 3.4 后端自动选择逻辑

```python
# memory_store.py 顶层（伪代码）

_SQLITE_AVAILABLE = False
try:
    import sqlite3
    _SQLITE_AVAILABLE = True
except ImportError:
    pass

if _SQLITE_AVAILABLE:
    _backend = SqliteBackend()
else:
    _backend = JsonFileBackend()

# 对外函数全部委托给 _backend
def add_entry(...):
    return _backend.add_entry(...)
```

**关键点**：`import sqlite3` 的失败被 `memory_store.py` 自己消化，不向外传播 ImportError。

## 四、惰性初始化

### 4.1 SqliteBackend

```python
class SqliteBackend:
    def __init__(self):
        self._conn = None        # 不在 __init__ 连接
        self._tables_ready = False

    def _get_conn(self):
        """首次调用时才建连接、建表"""
        if self._conn is None:
            os.makedirs(CONFIG_DIR, exist_ok=True)
            self._conn = sqlite3.connect(MEMORY_DB_PATH)
            self._conn.row_factory = sqlite3.Row
        if not self._tables_ready:
            self._ensure_tables(self._conn)
            self._tables_ready = True
        return self._conn
```

- 模块 import 时只做 `SqliteBackend()` 实例化 → 无 IO
- 第一次 `add_entry` / `search` 等操作时才触发 `_get_conn()` → 建连接 + 建表

### 4.2 JsonFileBackend

```python
class JsonFileBackend:
    JSON_PATH = os.path.join(CONFIG_DIR, "memory_entries.json")

    def __init__(self):
        self._data = None  # 惰性

    def _load(self):
        if self._data is not None:
            return self._data
        if os.path.isfile(self.JSON_PATH):
            with open(self.JSON_PATH, "r", encoding="utf-8") as f:
                self._data = json.load(f)
        else:
            self._data = {"entries": [], "next_id": 1}
        return self._data

    def _save(self):
        os.makedirs(CONFIG_DIR, exist_ok=True)
        with open(self.JSON_PATH, "w", encoding="utf-8") as f:
            json.dump(self._data, f, ensure_ascii=False, indent=2)
```

- 同样惰性：只在第一次需要数据时读文件

## 五、调用方防御链（三层）

| 层级 | 位置 | 防御方式 | 触发场景 |
|------|------|----------|----------|
| **第 1 层** | `memory_store.py` 内部 | sqlite3 import 失败 → 切换 JSON 后端 | 云端无 sqlite3 |
| **第 2 层** | `app_ui.py` 第 34-50 行 | try-except → `MEMORY_AVAILABLE = False` | memory_store.py 自身有 bug 导致 import 失败 |
| **第 3 层** | `crew_test.py` 第 852-858 行 | try-except → pass（仅 context 缺失，不崩溃） | 任何意外 |

### 5.1 各调用方现状与调整

| 文件 | 现有保护 | 需调整 |
|------|----------|--------|
| `app_ui.py:34-50` | try-except + MEMORY_AVAILABLE | 保持不变，作为最后防线 |
| `crew_test.py:852-858` | try-except ImportError → pass | 保持不变 |
| `agent_knowledge_service.py:23` | try-except ImportError → return "" | 保持不变 |
| `app_ui.py:1468` | try-except 内 from memory_store | 保持不变 |

重构后，第 2 层和第 3 层理论上不会再触发（因为 memory_store 自身不再抛 ImportError），但保留作为安全网。

## 六、哈希去重（统一实现）

两个后端共用同一套去重逻辑：

1. 计算 `SHA-256(content.strip())`
2. 查询已有条目是否存在相同 hash
3. 若存在 → 返回 `(existing_id, "skipped")`
4. 若不存在但 `source_type + source_id` 匹配 → 更新内容和 hash，返回 `(id, "updated")`
5. 均不匹配 → 新增，返回 `(new_id, "added")`

## 七、Agent 上下文注入链路（不变）

```
get_project_context_for_agent()
  ├── 1. load_project_memory() → 读 project_memory.md（纯文件，无 DB）
  ├── 2. load_fambase_modules_for_agent() → 读 fambase_modules.yaml（纯文件）
  ├── 3. [可选] load_agent_knowledge() → 读 agent_knowledge.md（纯文件）
  └── 4. [可选] get_recent_for_agent() → 从 memory_store 读条目
                                           ↑ try-except 保护
                                           ↑ 失败只丢失"近期需求"增量，不崩溃
```

**设计要点**：步骤 1-3 均为纯文件读取，与 sqlite3 完全无关。步骤 4 是增强层，有 try-except 兜底。即使 memory_store 完全不可用，Agent 仍有 project_memory.md + fambase_modules.yaml 作为基础上下文。

## 八、Gemini Context Cache（独立迭代，不阻塞）

- Context Cache 只需要一个「拼好的大文本」（来自 `get_project_context_for_agent()`）
- 与存储后端无关，无论 sqlite 还是 JSON，只要能拼出文本就能创建缓存
- 作为后续优化项单独迭代，不纳入本次「消除 sqlite3 风险」的范围

## 九、数据迁移兼容

| 场景 | 处理方式 |
|------|----------|
| 本地已有 memory.db，升级后继续用 sqlite | SqliteBackend 正常读取，无迁移 |
| 云端从 MEMORY_AVAILABLE=False 升级到 JSON 后端 | JsonFileBackend 发现无 memory_entries.json，初始化空库，用户重新导入 |
| 本地想从 sqlite 迁移到 JSON（可选） | 提供 `migrate_sqlite_to_json()` 工具函数，一次性导出 |
| JSON 后端积累数据后，环境支持 sqlite 了 | 提供 `migrate_json_to_sqlite()` 工具函数，一次性导入 |

迁移是可选的低优先级功能，不阻塞主逻辑。
