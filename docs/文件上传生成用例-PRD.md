# 文件上传生成用例 · PRD（与当前实现对齐）

> **文档用途**：定义「上传 .md / .docx / .xlsx 多文件 → 分析理解 → 产出问题点与新用例表」的产品方案，对齐当前 `run_upload_to_cases` 与 UI 实际行为。
>
> **最近更新**：2025-03-XX（补充 .docx 支持，结果展示为三块，并说明与历史记录的关系）

---

## 一、需求概述

### 1.1 背景

- 原「本地文件模式」为目录扫描 + Word(.docx) + 保存到工作目录。
- 新方案：**仅上传入口**，支持 **.md（需求文档）+ .docx（需求文档）+ .xlsx（既有测试用例）** 混合上传。
- 产出 **理解内容 + 问题点 + 新用例表** 三块内容，并提供 **Excel 下载** 按钮。

### 1.2 目标

- **输入**：用户一次上传多个 .md / .docx 和 .xlsx 文件。
- **处理**：解析 → 分析文档与既有用例 → 四 Agent 生成新用例。
- **输出**：
  1. 理解内容
  2. 问题点
  3. 新测试用例表（Markdown 表格 + Excel 下载）

---

## 二、方案变更对照

| 项目 | 原方案（已废弃） | 新方案 |
|------|------------------|--------|
| 输入方式 | 本地目录扫描 | **仅上传入口** |
| 导出方式 | 保存到工作目录、可配导出路径 | **仅 Excel 下载**，不配置导出路径 |
| 文件数量 | 单文件或目录内多选 | **支持多文件**（一次上传多个） |
| 文件类型 | .docx（Word） | **.md + .docx + .xlsx**（需求文档 + 既有测试用例） |
| 产出物 | 测试用例表 | **理解内容 + 问题点 + 新测试用例表（三块）** |

---

## 三、核心流程

```text
┌─────────────────────────────────────────────────────────────────────┐
│  1. 上传   │ 用户选择多个 .md / .docx（需求）和 .xlsx（既有用例），可混合   │
├─────────────────────────────────────────────────────────────────────┤
│  2. 解析   │ .md / .docx → 文本；.xlsx → 表格内容                      │
├─────────────────────────────────────────────────────────────────────┤
│  3. 生成   │ 经四 Agent 流水线生成理解内容、问题点、新测试用例表        │
├─────────────────────────────────────────────────────────────────────┤
│  4. 下载   │ 提供「下载 Excel」按钮（仅新测试用例表）                  │
└─────────────────────────────────────────────────────────────────────┘
```

## 四、范围与边界

### 4.1 在范围内

| 项目 | 说明 |
|------|------|
| 多文件上传 | `st.file_uploader(..., accept_multiple_files=True)`，类型为 `.md`、`.docx` 和 `.xlsx` |
| 解析逻辑 | .md 按 utf-8 读文本；.docx 用 python-docx 提取文本；.xlsx 用 `parse_test_cases_file` 读表格内容 |
| 分析产出 | 生成三块内容：理解内容 / 问题点 / 新测试用例表 |
| 用例生成 | 复用现有四 Agent 流水线，基于合并后的需求 + 既有用例上下文 |
| Excel 下载 | 通过 `st.download_button` 下载新用例表的 Excel，不暴露导出路径配置 |

### 4.2 不在范围内（本期）

| 项目 | 说明 |
|------|------|
| 导出路径配置 | 不提供工作目录、导出路径等配置项 |
| Quip / Google Sheets 导出 | 本功能仅在该入口支持 Excel 下载 |
| 目录扫描 | 不提供本地目录选择，仅上传 |

## 五、UI 与交互设计

### 5.1 上传入口

- **组件**：`st.file_uploader`，`accept_multiple_files=True`，`type=["md", "docx", "xlsx"]`。
- **说明文案**：支持上传多个 .md / .docx（需求文档）和 .xlsx（既有测试用例），可混合选择。
- **校验**：
  - 至少 1 个需求文档（.md 或 .docx）。
  - .xlsx 可选；若仅上传 .xlsx 则提示「至少需上传 1 个需求文档（.md 或 .docx）」。

### 5.2 解析与预览

- 上传成功后，展示简要预览：
  - .md / .docx：文件名 + 前 200 字摘要。
  - .xlsx：文件名 + 行数。
- 用户点击「开始生成」后进入分析 + 生成流程。

### 5.3 结果展示（三块布局）

| 区块 | 标题       | 内容说明 |
|------|------------|----------|
| 1    | 理解内容   | Agent 对需求文档的总结与关键要点理解 |
| 2    | 问题点     | 对需求中不清晰或有风险之处的梳理 |
| 3    | 新测试用例表 | Markdown 表格展示 + Excel 下载按钮 |

### 5.4 Excel 下载

- **仅** 提供 `st.download_button` 将新用例表导出为 `.xlsx`：
  - 文件名默认使用通用名（如 `测试用例.xlsx`），在历史记录模块中再统一按命名规则落盘。
- 不增加：导出路径输入、Quip 导出、Google Sheets 导出等。

---

## 六、数据流与接口

### 6.1 解析

```text
.md 文件   → read(encoding="utf-8") / "utf-8-sig" → 文本
.docx 文件 → python-docx 提取正文与表格文本 → 文本
.xlsx 文件 → parse_test_cases_file → 既有用例文本（| 分隔）
```

### 6.2 合并与传入流水线

- **需求文档**：所有 .md / .docx 内容合并（中间可添加 `---` 分隔符）。
- **既有用例**：所有 .xlsx 内容合并为文本（如 `|` 分隔），拼入 `project_context` 中。
- **demand**：以需求合并内容为主；既有用例作为「已知测试用例」附加说明传入 Agent。

### 6.3 分析阶段产出

- `run_upload_to_cases` 返回结构：
  - `understanding`: 理解内容（task2）。
  - `issues`: 问题点（task1）。
  - `cases_md`: 新测试用例表（task3 的 Markdown 表格）。
  - `excel_bytes`: 基于解析出的表格生成的 Excel 二进制，如无表格则为 `None`。

## 七、安全与防御

| 项 | 说明 |
|------|------|
| 文件类型白名单 | 仅接受 `.md`、`.docx`、`.xlsx`；拒绝 `.exe`、`.py`、`.bat` 等可执行文件 |
| 文件大小限制 | 单文件建议 \< 10MB，总上传 \< 50MB（已在解析侧限制） |
| Excel 注入 | 解析后的内容写入 Excel 时，使用 `_sanitize_cell_for_excel` 防公式注入 |
| 上传文件存储 | 上传文件仅在内存解析，不写入用户不可控路径；如开启历史记录，生成结果会写入受控的 workspace/output 目录 |

---

## 八、与现有模块的关系

| 模块        | 影响 |
|-------------|------|
| `crew_test` | 复用 `run_pipeline`、`_parse_markdown_tables`、`export_tables_to_excel_bytes`；通过 `existing_cases` 将既有用例并入上下文 |
| `parse_test_cases_file` | 复用 .xlsx 解析逻辑 |
| `app_ui`    | 在「生成用例」页中作为「上传文件」模式存在，结果展示三块并支持 Excel 下载；与 Quip 模式、粘贴模式共用底层流水线 |
| `run_history` | 上传模式生成成功后，会将结果写入历史记录 JSON，支持后续搜索、删除与下载 |

## 九、验收标准

| 编号 | 验收项 |
|------|--------|
| AC1  | 支持一次上传多个 .md / .docx / .xlsx 文件（可混合） |
| AC2  | .md / .docx 正确解析为文本，.xlsx 正确解析为表格内容 |
| AC3  | 结果展示分为三块：理解内容 / 问题点 / 新测试用例表 |
| AC4  | 提供「下载 Excel」按钮，可下载新用例表 |
| AC5  | 不提供导出路径、Quip、Sheets 等导出配置 |
| AC6  | 至少需 1 个 .md 或 .docx；.xlsx 可选；非法类型或空文件有明确提示 |

## 十、实现要点（供开发参考）

### 10.1 上传组件

```python
uploaded_files = st.file_uploader(
    "上传需求文档与既有用例",
    type=["md", "docx", "xlsx"],
    accept_multiple_files=True,
    help="支持 .md / .docx（需求文档）和 .xlsx（既有测试用例），可混合选择；单文件 < 10MB，总 < 50MB",
)
```

### 10.2 解析

- `.md`：`file.read().decode("utf-8-sig", errors="replace").strip()`
- `.docx`：使用 `_extract_text_from_docx` 统一提取文本。
- `.xlsx`：复用 `parse_test_cases_file`。

### 10.3 结果三块布局

```python
st.subheader("1. 理解内容")
st.markdown(result.get("understanding", "") or "*（无）*")

st.subheader("2. 问题点")
st.markdown(result.get("issues", "") or "*（无）*")

st.subheader("3. 新测试用例表")
st.markdown(result.get("cases_md", "") or "*（无）*")

excel_bytes = result.get("excel_bytes")
if excel_bytes:
    st.download_button(
        "📥 下载 Excel",
        data=excel_bytes,
        file_name="测试用例.xlsx",
        mime="application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
    )
```

### 10.4 与历史记录的衔接（补充）

- 上传模式与粘贴模式生成成功后：
  - 在 UI 中将本次输出（理解内容 / 问题点 / 新用例表）组合为文本摘要，写入受控目录下的 `.txt`。
  - 若存在 `excel_bytes`，同时写入受控目录下的 `.xlsx` 文件。
  - 通过 `run_history.add_run_record` 记录来源类型、需求标题、时间戳、Excel 路径与结果摘要，供历史列表使用。
