# 项目记忆 · RAG 与上下文管理 - 产品需求文档

> **文档用途**：定义项目记忆的定位、多 Agent 共享机制、自动更新与去重、历史记录与总结展示，为 4 个协作 Agent 搭建带状态的检索增强生成（RAG）与上下文管理系统。
>
> **创建日期**：2025-03-XX

---

## 一、需求概述

### 1.1 背景

- 当前项目记忆：需求文档历史 + 全回归用例导入，供 Agent 参考。
- 痛点：
  - 每次生成用例时，Agent 都会重新读取大量项目记忆文本，导致 Token 消耗高、响应慢；
  - 重复上传同一文档会重复入库，记忆库臃肿；
  - 导入历史缺少结构化总结，用户难以快速了解「导入了什么、核心内容是什么」。

### 1.2 目标

为 4 个协作 Agent 搭建**带状态的 RAG 与上下文管理系统**，实现：

1. **项目记忆定位**：作为整个项目的背景知识，支持导入多个全回归用例表格；
2. **多 Agent 共享且不重复消耗**：各 Agent 均能参考项目记忆，但不在每次生成时重复读取全文；
3. **自动更新与严格去重**：上传文档自动入库，重复内容（哈希相同）直接过滤；
4. **历史记录与自动总结**：展示导入记录，并为每次导入生成结构化总结（主要流程、核心内容）。

## 二、核心需求拆解

### 需求 1：项目记忆定位与数据来源

| 项目 | 说明 |
|------|------|
| 定位 | 项目记忆 = 整个项目的背景知识，供 4 个 Agent 在生成测试用例时作为长期推理背景 |
| 数据来源 | ① 多个全回归用例表格（通过上传或导入）；② 上传的 Word/Excel 文档；③ 粘贴的需求文档 |
| 使用方式 | 作为 Agent 的「上下文」注入，而非每次请求时重新拉取全文 |

### 需求 2：多 Agent 共享且不重复消耗

| 项目 | 说明 |
|------|------|
| 现状问题 | 每次生成用例时，4 个 Agent 都会收到完整的 project_context（几万字），导致 Token 消耗高、首字节延迟大 |
| 目标 | 各 Agent 均能参考项目记忆，但**不**在每次生成时重新读取全文 |
| 方案 | 使用 **Gemini Context Caching（上下文缓存）**：首次导入/更新后，将项目记忆打包调用 Gemini API 创建 Cached Content，返回 cache_name；后续多 Agent 请求时直接挂载 cache_name，模型无需重新阅读全文 |

### 需求 3：自动更新与严格去重

| 项目 | 说明 |
|------|------|
| 自动更新 | 用户上传/导入的文档，自动追加到项目记忆存储，无需手动「刷新知识库」 |
| 严格去重 | 使用 **SHA-256 哈希** 作为文件指纹，若新文档哈希已存在，直接跳过并提示「文件未变更，已跳过」 |
| 更新触发 | 若文件名相同但哈希变化，或为全新文件，则追加/替换存储，并触发 Gemini 缓存刷新（生成新 cache_name） |

### 需求 4：历史记录与自动总结

| 项目 | 说明 |
|------|------|
| 历史记录 | 在「项目记忆」Tab 下展示导入历史，每条记录包含：时间、操作类型、来源、文件名/标题 |
| 自动总结 | 每次新导入后，后台调用轻量模型（如 Gemini Flash）生成 150 字以内的中文总结，涵盖：主要业务模块、核心操作流程、增删改重点逻辑 |
| 展示形式 | 时间轴式动态列表，每条可展开查看总结与导入内容预览 |

## 三、架构设计（概念层）

### 3.1 数据分层

```
┌─────────────────────────────────────────────────────────────────┐
│  Metadata 层（元数据）                                            │
│  - 哈希值、总结文本、更新时间、来源、文件名                         │
│  - 供 Streamlit 快速读取和渲染列表，速度极快                        │
│  - 存储：memory_index 表或 memory_index.json                      │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Context 层（实体记忆）                                            │
│  - 原始文本流（全回归用例、需求文档等）                             │
│  - 有变动时推送到 Gemini Context Cache，生成 cache_name           │
│  - 存储：memory_entries 表 + project_memory.md 或等效             │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 流程概览

```
用户上传/导入文档
       │
       ▼
计算 SHA-256 哈希
       │
       ├── 哈希已存在 ──► 提示「文件未变更，已跳过」，不入库
       │
       └── 哈希不存在 ──► 入库（Metadata + Context）
                                │
                                ▼
                        触发 Librarian Agent 生成总结
                                │
                                ▼
                        刷新 Gemini Context Cache（生成新 cache_name）
                                │
                                ▼
                        4 个 Agent 后续请求挂载 cache_name，无需重复读全文
```

### 3.3 角色：Librarian Agent（图书管理员）

- **职责**：建档与审计，为新导入的文档生成结构化总结。
- **触发时机**：文件通过哈希校验、确认为新内容并入库后。
- **模型选择**：Gemini Flash 或 1.5 Flash-8B（成本低，仅需总结）。
- **Prompt 示例**：  
  「这是一份新上传的测试/需求文档。请用 150 字以内的中文，总结该文档主要涵盖的业务模块、核心操作流程以及增删改的重点逻辑。」
- **输出**：总结文本持久化到 Metadata 层，供 UI 展示。

## 五、验收标准

| 编号 | 验收项 |
|------|--------|
| AC1 | 支持从多个全回归用例表格导入，并作为项目记忆的一部分 |
| AC2 | 4 个 Agent 生成用例时能参考项目记忆，且不重复读取全文（通过 Context Caching 或等效机制） |
| AC3 | 重复上传相同内容（哈希一致）时，提示「文件未变更，已跳过」，不重复入库 |
| AC4 | 新导入后自动生成 150 字以内的中文总结，并展示在历史记录中 |
| AC5 | 项目记忆 Tab 展示时间轴式导入历史，每条可展开查看总结与内容预览 |
| AC6 | 项目记忆更新后，下次生成用例时自动使用最新缓存，无需用户手动刷新 |

## 六、非功能约束

- **安全**：哈希计算与存储不涉及敏感信息；Gemini API Key 通过 st.secrets 或 .env 读取。
- **性能**：Metadata 层查询需在 100ms 内完成；Context Cache 创建可异步，不阻塞用户操作。
- **兼容**：与现有「粘贴导入」「Excel/CSV 导入」流程兼容，新逻辑作为增强而非破坏性变更。
